# 移行ガイド：usememos/memos から本 Fork への移行

このドキュメントでは、上流 [usememos/memos](https://github.com/usememos/memos) と本 fork のデータベースレベルの互換性の違い、および[移行修復スクリプト](../scripts/migration-repair.sh)が行う修復内容について説明します。

## 互換性の概要

| 上流バージョン | 互換性 | 必要な操作 |
|---|---|---|
| v0.23.0 ~ v0.23.1 | 完全互換 | なし |
| v0.24.0 ~ v0.26.2 | 修復後に互換 | 移行修復スクリプトを実行 |

## 移行前の必須事項：データバックアップ

> **警告：移行前に必ずデータをバックアップしてください。この手順は省略できません。**
>
> 移行が失敗した場合や予期しない結果が生じた場合、バックアップがデータを復旧する唯一の方法です。バックアップがなければ、メモ、添付ファイル、設定が永久に失われる可能性があります。

**バックアップ手順：**

1. **実行中のサービスを停止する** — サービスが動作中に移行を行わないでください
2. **データディレクトリ全体をコピーする**（デフォルト：`~/.memos/`）：
   ```bash
   # Docker ユーザー
   docker stop memos
   cp -r ~/.memos/ ~/.memos-backup-$(date +%Y%m%d)/

   # バイナリユーザー
   # まず memos プロセスを停止してから：
   cp -r ~/.memos/ ~/.memos-backup-$(date +%Y%m%d)/
   ```
3. **MySQL/PostgreSQL ユーザー**はデータベースダンプも作成してください：
   ```bash
   # MySQL
   mysqldump -u user -p memos > memos-backup-$(date +%Y%m%d).sql

   # PostgreSQL
   pg_dump -U user memos > memos-backup-$(date +%Y%m%d).sql
   ```
4. **バックアップを検証する** — バックアップファイルが空でなく、適切なサイズであることを確認

バックアップの完了を確認してから移行を進めてください。

## 修復スクリプトの内容

修復スクリプトは2つのフェーズで実行されます。以下では各変更点の理由と修復方法を説明します。

### フェーズ1：上流の破壊的変更の取り消し

上流 usememos/memos が行ったが本 fork が採用しなかったスキーマ変更です。スクリプトがこれらを検出して取り消します。

#### 上流 v0.26：`resource` テーブルが `attachment` にリネーム

- **上流マイグレーションファイル**：`0.26/00__rename_resource_to_attachment.sql`
- **上流が行ったこと**：`resource` テーブルを `attachment` にリネームし、新しい名前でインデックスを再作成、その後のマイグレーションですべてのインデックスを削除。
- **なぜ問題になるか**：本 fork はすべてのファイル添付操作に `resource` テーブルを使用しています。テーブルがリネームされると、すべてのリソースクエリが失敗します。
- **スクリプトの修復内容**：`attachment` テーブルが存在し `resource` が存在しない場合を検出し、リネームを元に戻します。必要なインデックスを再作成します（SQLite のみ — MySQL と PostgreSQL では本 fork は明示的な resource インデックスを使用しません）。

#### 上流 v0.26：`memo_organizer` テーブルが削除

- **上流マイグレーションファイル**：`0.26/01__drop_memo_organizer.sql`
- **上流が行ったこと**：v0.24 で、メモのピン留め状態を `memo_organizer` テーブルから `memo` テーブルの新しい `pinned` カラムに移行。その後 v0.26 で `memo_organizer` を完全に削除。
- **なぜ問題になるか**：本 fork はユーザーごとのメモピン留め機能に `memo_organizer` を使用しています。このテーブルがないと、ピン留め機能が動作しません。
- **スクリプトの修復内容**：欠落している場合は `memo_organizer` テーブルを再作成。テーブルが新たに作成された場合（上流によって削除されていたことを意味する）かつ `memo.pinned` カラムが存在する場合（上流 v0.24 由来）、ピン留めデータを `memo_organizer` に復元します。

#### 上流 v0.26：`HOST` ロールが `ADMIN` に変更

- **上流マイグレーションファイル**：`0.26/03__alter_user_role.sql` + `0.26/04__migrate_host_to_admin.sql`
- **上流が行ったこと**：ユーザーロールシステムから `HOST` ロールを削除し、HOST 制約なしでユーザーテーブルを再作成、すべての HOST ユーザーを ADMIN に変換。
- **なぜ問題になるか**：本 fork は権限システム全体で `HOST` をプライマリ管理者ロールとして使用しています。HOST ユーザーがいないと、ワークスペース設定、IDP 設定などの管理者操作が実行できません。
- **スクリプトの修復内容**：HOST ロールを持つユーザーがいるか確認。いない場合、最も小さい ID を持つ ADMIN ユーザー（通常はインスタンスの最初の作成者）に HOST を割り当てます。

#### 上流 v0.25：`webhook` テーブルが削除

- **上流マイグレーションファイル**：`0.25/00__remove_webhook.sql`
- **上流が行ったこと**：`webhook` テーブルを完全に削除し、Webhook 機能を廃止。
- **なぜ問題になるか**：本 fork は Webhook サポートを維持しています。テーブルがないと、Webhook 関連の API 呼び出しが失敗します。
- **スクリプトの修復内容**：欠落している場合は `webhook` テーブルを再作成。注意：上流インスタンスで以前設定されていた Webhook データは、上流がテーブルを削除した際に失われています。手動で再設定が必要です。

### フェーズ2：Fork 固有テーブルの作成

本 fork が追加した、上流には存在しないテーブルです。Fork のマイグレーションシステムは、マイグレーション履歴により高いバージョン番号が存在する場合（上流由来）、作成をスキップすることがあります。

#### Fork v0.24：`tag` テーブル

- **Fork マイグレーションファイル**：`0.24/01__tag.sql`
- **機能説明**：絵文字アイコンとタグピン留めをサポートするタグ管理機能を提供。
- **なぜ欠落する可能性があるか**：Fork の 0.24 マイグレーション内容は上流の 0.24 と異なります。上流の 0.24 が既に適用されている場合、fork のマイグレーターはスキップし、`tag` テーブルが作成されません。
- **スクリプトの修復内容**：存在しない場合、`tag` テーブルとそのインデックスを作成。

#### Fork v0.25：メモレビュー関連テーブル

- **Fork マイグレーションファイル**：`0.25/00__review.sql` + `0.25/01__memo_review_session_cache.sql`
- **機能説明**：セッショントラッキングとキャッシュを備えたメモの間隔復習システムを提供。
- **作成されるテーブル**：
  - `memo_review_session` — ユーザーごとのレビューセッションを追跡
  - `memo_review` — セッション内の個別メモレビュー記録
  - `memo_review_session_cache` — 進行中のレビューセッション状態をキャッシュ
- **なぜ欠落する可能性があるか**：tag テーブルと同じバージョン番号の衝突問題。上流の 0.25 マイグレーション（webhook 削除）と fork の 0.25（レビューテーブル作成）は内容が全く異なります。
- **スクリプトの修復内容**：存在しない場合、上記3つのテーブルとそのインデックスを作成。

## 移行完了後

修復スクリプト実行後：

1. **サービスを起動**してエラーがないことを確認
2. **メモを確認** — 内容が完全に表示されることを確認
3. **ピン留めメモを確認** — ピン留めしていたメモが引き続きピン留めされていることを確認
4. **Webhook を再設定** — 上流 v0.25+ で Webhook を使用していた場合、再設定が必要です（上流がテーブルを削除した際にデータが失われています）
5. **管理者アクセスを確認** — 管理者ユーザーが正しい権限を持っていることを確認
